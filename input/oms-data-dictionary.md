# OMS Data Dictionary

Plain-language definitions for every field in every table across the five database modules.
Written for new team members with no prior knowledge of the Order Management System.

---

## How to read this document

Each table entry has:
1. **A one-line plain description** of what the table represents in the real world.
2. **A field glossary** explaining what each column stores and why it exists.

**Column name conventions used throughout:**
| Suffix | Meaning |
|---|---|
| `_id` (PK) | A unique internal identifier generated by OMS. Never changes, never reused. |
| `_id` (FK) | A reference to a row in another table. |
| `_at` | A point in time (timestamp with timezone). |
| `_by` | A person or system that performed an action. |
| `_flag` / `is_` / `_pending` | A yes/no switch. |
| `source_` | A value that came from an **external** system (POS, ERP, payment gateway). |

## Module 1 — Order (schema: `orders`)

These tables track everything about a customer's order from the moment it is placed until it is paid and closed.

---

### `orders` — the core order record

The master record for a single customer order. Every other table in this module links back here.

| Column | Type | Plain meaning |
|---|---|---|
| `order_id` | UUID PK | OMS's own permanent ID for this order. Used internally in all JOIN operations. |
| `order_number` | varchar UK | The human-readable code shown to staff, drivers, and customers — e.g. `ORD-001`. Unique and never changes. |
| `source_order_id` | varchar UK | The order ID from whichever external system created this order (e-commerce platform, POS, app). Lets OMS trace back to the origin system. |
| `channel_type` | varchar | Where the order came from: `Web`, `App`, `POS`, `CallCenter`, etc. Determines which business rules apply. |
| `business_unit` | varchar | Which brand or business group owns this order (e.g. `TOPS`, `Central Food`). Used for reporting and routing. |
| `store_id` | varchar FK | Which store or distribution centre (DC) will pick, pack, and dispatch this order. References `config.store_locations`. |
| `order_date` | timestamptz | The exact moment the customer placed the order. |
| `status` | varchar | The current lifecycle stage of the order. Moves in sequence: `Pending` → `BookingConfirmed` → `PickStarted` → `PickConfirmed` → `Packed` → `OutForDelivery` → `Delivered` → `Invoiced` → `Paid`. Can also be `OnHold`, `Cancelled`, or `Returned`. |
| `pre_hold_status` | varchar | The status the order was in immediately before it was put on hold. When the hold is lifted, OMS restores the order to this status rather than guessing. |
| `hold_reason` | varchar | Why the order is currently paused — e.g. `AddressUnverified`, `PackageDamaged`, `FraudSuspected`. Only populated when `status = 'OnHold'`. |
| `fulfillment_type` | varchar | How the order will reach the customer: `Delivery` (driver brings it to the address), `Express` (same-day fast delivery), or `ClickAndCollect` (customer picks up at the store). |
| `payment_method` | varchar | How the customer chose to pay: `CreditCard`, `QRCode`, `COD` (cash on delivery), `Wallet`, etc. |
| `substitution_flag` | bool | `true` if the customer agreed to accept a replacement product when their requested item is out of stock. `false` means the line must be cancelled if unavailable. |
| `pos_recalc_pending` | bool | A safety lock set to `true` after picking while OMS waits for the POS system to recalculate the final price with promotions applied. **The order cannot be packed until this is `false`.** Prevents charging the wrong amount. |
| `created_at` | timestamptz | When this order record was first created in OMS. |
| `updated_at` | timestamptz | The last time any field on this record was changed. |
| `created_by` | varchar | The user or system that created the order. |
| `updated_by` | varchar | The user or system that last modified the order. |

### `order_lines` — individual items inside an order

One row per product line. An order for 3 different products has 3 rows here.

| Column | Type | Plain meaning |
|---|---|---|
| `order_line_id` | UUID PK | OMS's unique ID for this product line. |
| `order_id` | UUID FK | The order this line belongs to. References `orders`. |
| `sku` | varchar | Product code (Stock Keeping Unit). The unique identifier for this product variant — e.g. `APPLE-1KG`, `MILK-UHT-1L`. |
| `product_name` | nvarchar | The product's display name copied from the product catalogue at order creation — e.g. `Fresh Apple 1 kg bag`. Preserved so the name in the order never changes even if the catalogue is later updated. |
| `barcode` | varchar | The product barcode printed on the packaging. Pickers scan this to confirm they picked the right item. |
| `requested_amount` | decimal | How many units the customer ordered. |
| `picked_amount` | decimal | How many units were actually taken from the shelf. May be less than `requested_amount` if the item was short or out of stock. Updated when WMS sends the `PickConfirmed` webhook. |
| `unit_of_measure` | varchar | The counting unit: `Each` (individual items), `Kg` (by weight), `Pack`, etc. |
| `original_unit_price` | decimal | Price per unit at the time the order was placed. Kept unchanged as a permanent record. The final charged price may differ after POS applies promotions — see `payment.order_line_amounts`. |
| `currency` | varchar | Currency of the price (e.g. `THB`). |
| `status` | varchar | Status of this individual line: `Pending`, `Picked`, `Substituted`, `Cancelled`. Useful when only some lines of an order are affected. |
| `is_substitute` | bool | `true` if this line was **added** as a replacement for another line that could not be fulfilled. |
| `created_at` | timestamptz | When this line was added to the order. |
| `updated_at` | timestamptz | When this line was last changed. |

### `order_line_substitutions` — replacement product proposals

When a picker cannot find the exact item and `substitution_flag` is `true` on the order, they propose a replacement. This table records what was offered and whether the customer approved.

| Column | Type | Plain meaning |
|---|---|---|
| `substitution_id` | UUID PK | Unique ID for this substitution record. |
| `order_line_id` | UUID FK | The original order line that could not be fulfilled as requested. |
| `substitute_sku` | varchar | SKU of the proposed replacement product. |
| `substitute_product_name` | nvarchar | Display name of the replacement product. |
| `substitute_unit_price` | decimal | Price of the replacement product per unit. |
| `substituted_amount` | decimal | Quantity of the replacement product offered. |
| `customer_approved` | bool | `true` = customer accepted the swap. `false` = customer declined; the original line is cancelled. |
| `created_at` | timestamptz | When the substitution was proposed. |

### `order_packages` — physical packages / boxes

One row per sealed box or bag that leaves the warehouse. An order may span multiple packages.

| Column | Type | Plain meaning |
|---|---|---|
| `package_id` | UUID PK | OMS's unique ID for this package. |
| `order_id` | UUID FK | The order this package belongs to. |
| `tracking_id` | varchar UK | The tracking code printed on the label stuck to the package. Customers and drivers use this to track the delivery — e.g. `TRK-2024-001`. |
| `carrier_package_id` | varchar | The delivery company's own reference number for this package (may differ from `tracking_id`). |
| `third_party_logistic` | varchar | The name of the logistics company handling delivery — e.g. `Flash Express`, `Kerry`. |
| `vehicle_type` | varchar | Type of vehicle assigned for this delivery: `Motorcycle`, `Van`, `Truck`. Affects route planning. |
| `status` | varchar | Current delivery status of this physical package: `Packed`, `OutForDelivery`, `Delivered`, `ReturnedToWarehouse`. |
| `package_weight` | decimal | Total weight of the sealed package in kilograms. Used by TMS for route loading calculations. |
| `delivery_note_number` | varchar | Reference number on the paper delivery note placed inside the package. Used as proof of shipment content. |
| `created_at` | timestamptz | When the package was created in OMS. |
| `updated_at` | timestamptz | When the package record was last changed. |

### `order_package_lines` — which items are in which package

A join table. Each row says: "order line X is packed inside package Y."
One package contains many lines; each line belongs to exactly one package.

| Column | Type | Plain meaning |
|---|---|---|
| `id` | UUID PK | Unique ID for this assignment row. |
| `package_id` | UUID FK | The package. References `order_packages`. |
| `order_line_id` | UUID FK | The order line inside that package. References `order_lines`. |

| Column | Type | Plain meaning |
|---|---|---|
| `address_id` | UUID PK | Unique ID for this address record. |
| `order_id` | UUID FK | The order this address belongs to. |

| `last_name` | nvarchar | Recipient's last name. |
| `company_name` | nvarchar | Company name for corporate deliveries or tax invoices. |
| `address1` | nvarchar | Primary address line — building number, street, floor. |
| `address2` | nvarchar | Additional address detail — landmark, room number. |
| `subdistrict` | nvarchar | Subdistrict (แขวง / ตำบล in Thai). |
| `district` | nvarchar | District (เขต / อำเภอ in Thai). |
| `province` | nvarchar | Province (จังหวัด). |
| `postal_code` | varchar | Postal / ZIP code. |
| `country_code` | varchar | ISO 2-letter country code — e.g. `TH`. |
| `mobile_phone` | varchar | Phone number the driver calls if they cannot find the address. |
| `email` | varchar | Email used for delivery notifications and invoices. |
| `latitude` | varchar | GPS latitude of the delivery point. Used by TMS for map routing. |
| `longitude` | varchar | GPS longitude of the delivery point. |
| `tax_id` | varchar | Tax ID number for full tax invoices. Required by corporate customers and government entities. |
| `created_at` | timestamptz | When the address was saved. |

### `order_customers` — the customer identity on an order

Links an order back to the customer record in the external loyalty or membership system.

| Column | Type | Plain meaning |
|---|---|---|
| `order_customer_id` | UUID PK | Unique ID for this record. |
| `order_id` | UUID FK | The order this customer is linked to. |
| `source_customer_id` | varchar | The customer's ID in the system that placed the order (loyalty app, POS, website). |
| `member_type` | varchar | Customer tier at the time of the order — e.g. `Gold`, `Silver`, `Guest`. Captured at order time so tier changes later don't alter historical orders. |
| `member_id` | varchar | Loyalty card / membership number. Used for earning and redeeming points. |
| `created_at` | timestamptz | When the customer was linked to this order. |

### `delivery_slots` — the time window booked for delivery

When a customer books a delivery, they pick a time window (e.g. 18:00–20:00). This table stores that booking.

| Column | Type | Plain meaning |
|---|---|---|
| `slot_id` | UUID PK | Unique ID for this slot booking. |
| `order_id` | UUID FK | The order this slot is for. |
| `store_id` | varchar FK | Which store's delivery calendar this slot was booked from. |
| `scheduled_start` | timestamptz | Start of the delivery window (e.g. 2024-05-06 18:00:00+07). |
| `scheduled_end` | timestamptz | End of the delivery window (e.g. 2024-05-06 20:00:00+07). |
| `created_at` | timestamptz | When the slot was booked. |
| `updated_at` | timestamptz | When the booking was last changed (e.g. customer rescheduled). |

### `order_holds` — hold events log

Every time an order is put on hold or released, a row is added here. Gives a full history of pauses.

| Column | Type | Plain meaning |
|---|---|---|
| `hold_id` | UUID PK | Unique ID for this hold event. |
| `order_id` | UUID FK | The order that was held. |
| `hold_reason` | varchar | Why the order was paused — e.g. `PackageDamaged`, `AddressUnverified`, `FraudSuspected`. |
| `held_at` | timestamptz | When the hold was applied. |
| `released_at` | timestamptz | When the hold was lifted. `null` if the order is still on hold. |
| `held_by` | varchar | The user or system that placed the hold. |
| `released_by` | varchar | The user or system that released the hold. |

### `order_outbox` — messages waiting to be sent to external systems

Instead of calling WMS, TMS, or POS directly (which might fail), OMS writes a message here **inside the same database transaction** as the domain change. A background worker then reads and sends it. This guarantees the message is never lost even if the external system is temporarily unavailable.

| Column | Type | Plain meaning |
|---|---|---|
| `outbox_id` | UUID PK | Unique ID for this queued message. |
| `order_id` | UUID FK | The order this message is about. |
| `event_type` | varchar | What kind of message this is — e.g. `OrderCreatedEvent`, `PickConfirmedEvent`, `PackagesAssignedEvent`. The receiver uses this to decide how to process it. |
| `event_payload` | jsonb | The full message content as JSON, exactly as it will be sent to the external system. |
| `status` | varchar | Whether the message has been sent: `Pending` (not yet sent), `Published` (sent successfully), `Failed` (gave up after max retries). |
| `retry_count` | int | How many times the worker has tried and failed to send this message. |
| `created_at` | timestamptz | When the message was created and queued. |
| `published_at` | timestamptz | When the message was successfully sent. `null` until successful. |
| `next_retry_at` | timestamptz | When the worker should try again after a failed attempt. Uses exponential backoff. |

---

### `order_status_history` — complete audit trail of status changes

Every time an order changes status, a new row is added here. You can reconstruct the full lifecycle of any order from this table.

| Column | Type | Plain meaning |
|---|---|---|
| `history_id` | UUID PK | Unique ID for this history entry. |
| `order_id` | UUID FK | The order whose status changed. |
| `from_status` | varchar | The status **before** the change. |
| `to_status` | varchar | The status **after** the change. |
| `changed_by` | varchar | Who or what triggered the change — a staff user ID, a system name, or a webhook source (e.g. `WMS`, `TMS`). |
| `detail` | varchar | Human-readable explanation of why the change happened — e.g. `"All 5 lines picked, no substitutions"`. Useful for investigations. |
| `changed_at` | timestamptz | Exact time the change was recorded. |

---

### `order_webhook_logs` — log of messages received from external systems

Every time WMS, TMS, or POS sends a callback to OMS, a row is recorded here. Used for debugging, audit, and the order timeline view.

| Column | Type | Plain meaning |
|---|---|---|
| `webhook_log_id` | UUID PK | Unique ID for this log entry. |
| `order_id` | uuid | The order this message relates to. May be empty for PO/transfer webhooks that arrive before an order exists. |
| `source_system` | varchar | Which external system sent the message: `WMS`, `TMS`, or `POS`. |
| `event_type` | varchar | The name of the event sent — e.g. `PickConfirmed`, `PackageDelivered`, `RecalculationResult`. |
| `detail` | varchar | A human-readable summary of what the webhook reported — e.g. `"5/5 lines picked, no substitutions"`. |
| `received_at` | timestamptz | When OMS received and processed this webhook. |

## Module 2 — Payment (schema: `payment`)

These tables record how an order is priced, taxed, invoiced, and paid.

---

### `order_payments` — the top-level payment record

The single payment record for an order. Think of it as the "bill" for an order.

| Column | Type | Plain meaning |
|---|---|---|
| `payment_id` | UUID PK | OMS's unique ID for this payment. |
| `order_id` | uuid | The order being paid. |
| `source_payment_id` | varchar UK | The payment reference from the external payment gateway (e.g. credit card processor). Used for reconciliation. |
| `currency` | varchar | Currency of the payment (e.g. `THB`). |
| `status` | varchar | Payment status: `Pending`, `Paid`, `Failed`, `Refunded`. |
| `payment_date` | timestamptz | When payment was confirmed by the payment gateway. |
| `created_at` | timestamptz | When the payment record was created. |
| `updated_at` | timestamptz | When the payment record was last changed. |

---

### `payment_transactions` — individual charge events

One row per charge attempt. A single payment may be split across multiple methods (e.g. partly by wallet, partly by card).

| Column | Type | Plain meaning |
|---|---|---|
| `transaction_id` | UUID PK | Unique ID for this transaction. |
| `payment_id` | UUID FK | The parent payment this transaction belongs to. |
| `payment_method` | varchar | How this portion was charged: `CreditCard`, `QRCode`, `Wallet`, `COD`. |
| `payment_sub_method` | varchar | More specific detail — e.g. card brand `Visa`, `Mastercard`, or wallet type `TrueMoney`. |
| `amount` | decimal | Amount charged in this single transaction. |
| `currency` | varchar | Currency. |
| `source_transaction_id` | varchar | The transaction reference ID from the payment gateway. Used for refund and dispute resolution. |
| `bill_address_ref` | varchar | Reference to the billing address used for this charge — links to the issuing bank address check. |
| `created_at` | timestamptz | When this transaction was recorded. |

---

### `invoices` — official tax invoices

An invoice is generated when the order is delivered. It is the legal document the customer and the business need for tax purposes.

| Column | Type | Plain meaning |
|---|---|---|
| `invoice_id` | UUID PK | Unique ID for this invoice. |
| `order_id` | uuid | The order this invoice covers. |
| `invoice_number` | varchar UK | The official tax invoice number printed on the document — e.g. `INV-2024-001`. Sequential and unique. |
| `invoice_type` | varchar | Type of document: `TaxInvoice` (full details for corporate customers) or `SimplifiedReceipt` (for consumers). |
| `total_amount` | decimal | Final invoiced total in the smallest currency unit. |
| `currency` | varchar | Currency. |
| `status` | varchar | Invoice status: `Generated`, `Sent` (emailed to customer), `Cancelled`. |
| `generated_at` | timestamptz | When the invoice was created — typically triggered by the `PackageDelivered` event. |
| `created_at` | timestamptz | When the record was saved. |

---

### `credit_notes` — refund and adjustment notes

A credit note is the opposite of an invoice. It reduces the amount a customer owes — issued when items are returned or a price error is corrected.

| Column | Type | Plain meaning |
|---|---|---|
| `credit_note_id` | UUID PK | Unique ID for this credit note. |
| `order_id` | uuid | The original order being credited. |
| `invoice_id` | UUID FK | The invoice being partially or fully reversed. |
| `credit_note_number` | varchar UK | Official reference number on the credit note document — e.g. `CN-RET-001`. |
| `amount` | decimal | The amount being credited back to the customer. |
| `currency` | varchar | Currency. |
| `reason` | varchar | Why the credit was issued: `Return`, `PriceAdjustment`, `DamagedGoods`. |
| `source` | varchar | What triggered the credit: `ReturnRequest`, `ManualAdjustment`, `PickShortfall`. |
| `status` | varchar | Status: `Issued`, `Applied` (offset against the customer's account), `Cancelled`. |
| `created_at` | timestamptz | When the credit note was created. |
| `updated_at` | timestamptz | When it was last changed. |

---

### `order_line_amounts` — price calculations per recalculation round

After picking, the POS system recalculates the actual charge based on real picked quantities and live promotions. A new row is inserted for each line in each recalculation round. **Multiple rows per line are normal** — earlier rounds are kept as history.

| Column | Type | Plain meaning |
|---|---|---|
| `amount_id` | UUID PK | Unique ID for this calculation record. |
| `order_line_id` | uuid | The order line this calculation is for. |
| `recalc_round` | int | Which recalculation this is. Round 1 = original price at order creation. Round 2 = price after picking. Higher rounds = subsequent adjustments. |
| `trigger_event` | varchar | What caused this recalculation — e.g. `OrderCreated`, `PickConfirmed`. Useful for tracing why the price changed. |
| `recalculated_unit_price` | decimal | Price per unit after promotions are applied. This is what the customer is actually charged. |
| `currency` | varchar | Currency. |
| `gross_amount` | decimal | Total for this line before tax. |
| `net_amount` | decimal | Total for this line after tax (gross − tax). |
| `unit_gross_amount` | decimal | Per-unit price before tax. |
| `unit_net_amount` | decimal | Per-unit price after tax. |
| `recalculated_at` | timestamptz | When POS performed this recalculation. |
| `created_at` | timestamptz | When the record was saved in OMS. |

---

### `order_line_taxes` — tax breakdown per amount record

Breaks down the tax portion of a line amount into individual tax components (VAT, excise tax, etc.).

| Column | Type | Plain meaning |
|---|---|---|
| `tax_id` | UUID PK | Unique ID for this tax row. |
| `amount_id` | UUID FK | The price calculation record this tax belongs to. References `order_line_amounts`. |
| `tax_type` | varchar | The tax category: `VAT`, `ExciseTax`, `WithholdingTax`. |
| `tax_description` | varchar | Human-readable name — e.g. `Value Added Tax 7%`. |
| `amount` | decimal | The tax amount in currency for this line. |
| `rate` | decimal | The tax rate as a decimal — e.g. `0.07` for 7% VAT. |
| `created_at` | timestamptz | When the record was created. |

---

| `currency` | varchar | Currency. |

### `order_promotions` — discounts applied to an order

One row per promotion applied. An order can have multiple promotions (e.g. a member discount and a product-specific offer applied at the same time).

| Column | Type | Plain meaning |

## Module 3 — Returns (schema: `returns`)

These tables track a customer's request to return items and the full process of getting them back to the warehouse, inspected, and refunded.

---

### `returns` — the return request

One row per return. The top-level record for a customer return from order creation through to refund.

| Column | Type | Plain meaning |
|---|---|---|
| `return_id` | UUID PK | OMS's unique ID for this return. |
| `order_id` | uuid | The original order the customer is returning items from. |
| `return_order_number` | varchar UK | Human-readable return reference shown to staff and customers — e.g. `RET-001`. |
| `invoice_id` | varchar | The invoice being partially reversed by this return. |
| `credit_note_id` | varchar | The credit note that will be issued to the customer when the return is processed. |
| `status` | varchar | Current stage of the return: `Requested` → `PickupScheduled` → `PickedUp` → `Received` → `Inspected` → `PutAway` → `Refunded`. |
| `goods_receive_no` | varchar | The warehouse receipt number issued when the returned goods physically arrive at the warehouse dock. |
| `return_reason` | varchar | Why the customer is returning the items — e.g. `WrongItem`, `Damaged`, `ChangeOfMind`, `QualityIssue`. |
| `requested_at` | timestamptz | When the customer submitted the return request. |
| `pickup_scheduled_at` | timestamptz | When a driver is scheduled to collect the return from the customer. |
| `picked_up_at` | timestamptz | When the driver actually collected the items from the customer. |
| `received_at` | timestamptz | When the warehouse physically received the returned goods. |
| `inspected_at` | timestamptz | When warehouse staff finished inspecting all returned items. |
| `put_away_at` | timestamptz | When the returned items were shelved, sent for repair, or routed for disposal. |
| `refunded_at` | timestamptz | When the refund was processed and sent to the customer. |
| `created_at` / `updated_at` | timestamptz | Standard audit timestamps. |
| `created_by` / `updated_by` | varchar | Standard audit user fields. |

---

### `return_items` — individual items inside a return

One row per product being returned. A customer returning 3 different items from one order creates 3 rows here.

| Column | Type | Plain meaning |
|---|---|---|
| `return_item_id` | UUID PK | Unique ID for this returned item. |
| `return_id` | UUID FK | The return request this item belongs to. |
| `order_line_id` | uuid | The original order line this item came from. Used to calculate the refund amount. |
| `sku` | varchar | Product code. |
| `product_name` | nvarchar | Product display name. |
| `barcode` | varchar | Barcode for scanning at the warehouse dock when the return arrives. |
| `quantity` | decimal | How many units of this product the customer is returning. |
| `unit_of_measure` | varchar | The counting unit (Each, Kg, etc.). |
| `unit_price` | decimal | Price per unit from the original order. Used to calculate the refund. |
| `currency` | varchar | Currency. |
| `item_reason` | varchar | Return reason specific to this item (may differ from the overall return reason if a mixed return). |
| `condition` | varchar | Physical condition assessed by warehouse staff during inspection: `Resellable` (can go straight back on the shelf), `Repairable` (needs fixing first), `Dispose` (write off as a loss). |

| `assigned_sloc` | varchar | The storage location code where the item was placed after inspection — e.g. `B-05`, `REPAIR-BAY-2`, `DISPOSAL`. |
| `payment_method` | varchar | How the refund for this particular item will be paid to the customer. |
| `inspected_at` | timestamptz | When this item was inspected. |
| `put_away_at` | timestamptz | When this item was shelved or disposed. |
| `created_at` / `updated_at` | timestamptz | Standard audit timestamps. |

| `quantity` | decimal | Number of units put away in this action. |
| `performed_by` | varchar | The warehouse staff member who physically placed the item. |
| `performed_at` | timestamptz | Exact time the put-away was confirmed. |

## Module 4 — Configuration (schema: `config`)

These tables hold the business rules and master data that all other modules refer to. Changes here affect how the entire system behaves.

---

### `store_locations` — every store and distribution centre

One row per physical location where OMS is active. Every order, PO, and transfer references a row here.

| Column | Type | Plain meaning |
|---|---|---|
| `store_id` | UUID PK | OMS's unique ID for this location. |
| `source_bu` | varchar | Business unit code from the legacy system (for migration mapping). |
| `source_loc` | varchar | Location code from the legacy system (for migration mapping). |
| `store_name` | nvarchar | Display name of the location — e.g. `Central DC`, `Store A — Sukhumvit`. |
| `business_unit` | varchar FK | Which brand group operates this store. References `business_units`. |
| `is_active` | bool | `true` if this store is currently accepting orders. |
| `is_rolled_out` | bool | `true` if the OMS is live at this store. `false` means the store uses the old system. |
| `address1` → `longitude` | various | Physical address and GPS coordinates of the store, used for routing and customer-facing maps. |
| `mobile_phone` | varchar | Store contact number for operations queries. |
| `email` | varchar | Store contact email. |
| `created_at` / `updated_at` | timestamptz | Standard audit timestamps. |

---

### `business_units` — brands and business groups

| Column | Type | Plain meaning |

### `rollout_policies` — controls when OMS goes live at a store

| Column | Type | Plain meaning |

### `fulfillment_routing_rules` — automatic routing decisions

When an order arrives, OMS uses these rules to decide automatically how to handle it — does it need a delivery slot? Does it need a TMS shipment? Different channels and fulfillment types follow different rules.

| Column | Type | Plain meaning |
|---|---|---|

| `created_at` | timestamptz | When the rule was created. |

## Module 5 — Inbound (schema: `inbound`)

These tables track goods **arriving at the warehouse**: supplier deliveries (Purchase Orders), stock movements between stores (Transfer Orders), and damaged packages returned by drivers.

---

### `purchase_orders` — buying goods from a supplier

A Purchase Order (PO) is raised by the buying team when they need to restock a store from a supplier. The supplier delivers against this PO and the warehouse checks it in.

| Column | Type | Plain meaning |
|---|---|---|
| `purchase_order_id` | UUID PK | OMS's unique ID for this PO. |
| `po_number` | varchar UK | Human-readable PO number printed on delivery paperwork — e.g. `PO-001`. Shown to the supplier and warehouse staff. |
| `supplier_id` | uuid | Who is delivering the goods. References the external supplier registry (not stored in OMS). |
| `store_id` | UUID FK | Which store or DC is receiving this delivery. References `config.store_locations`. |
| `status` | varchar | Current stage: `Created` (PO raised, not yet arrived) → `PartiallyReceived` (some goods arrived) → `FullyReceived` (all goods arrived) → `Closed` (goods put away, stock updated). |
| `goods_receive_no` | varchar | The warehouse goods receipt number (GRN) issued when the goods arrive at the dock. Set by the WMS. |
| `created_at` / `updated_at` | timestamptz | Standard audit timestamps. |
| `updated_by` | varchar | Who last updated this PO. |

---

### `purchase_order_lines` — individual products on a purchase order

One row per product included in the PO.

| Column | Type | Plain meaning |
|---|---|---|
| `po_line_id` | UUID PK | Unique ID for this line. |
| `purchase_order_id` | UUID FK | The PO this line belongs to. |
| `sku` | varchar | Product code being ordered from the supplier. |
| `ordered_qty` | int | How many units the business ordered. |
| `received_qty` | int | How many units actually arrived at the dock. May be less than `ordered_qty` if the supplier short-shipped — this creates a `PartiallyReceived` status on the PO. |
| `unit_cost` | decimal | Cost price per unit agreed with the supplier. Used for inventory valuation and margin calculation. |
| `currency` | varchar | Currency of the cost price. |
| `condition` | varchar | Physical state of the goods after inspection at put-away: `Resellable` (shelved normally), `Repairable` (damaged but fixable), `Dispose` (written off). |
| `sloc` | varchar | The storage location code where the goods were physically placed on the shelf — e.g. `A-12`. Set when the WMS confirms put-away. |
| `received_at` | timestamptz | When these goods were physically received at the warehouse dock. |
| `put_away_at` | timestamptz | When the goods were placed on the shelf and made available for picking. Stock becomes available in WMS after this point. |

---

### `transfer_orders` — moving stock between stores

A Transfer Order (TO) is raised when a store needs to replenish stock from another store or the central DC, rather than waiting for a supplier delivery.

| Column | Type | Plain meaning |
|---|---|---|
| `transfer_order_id` | UUID PK | OMS's unique ID for this transfer. |
| `transfer_number` | varchar UK | Human-readable reference — e.g. `TR-001`. Used on documents and labels. |
| `source_store_id` | UUID FK | The store or DC **sending** the goods. References `config.store_locations`. |
| `dest_store_id` | UUID FK | The store or DC **receiving** the goods. References `config.store_locations`. |
| `status` | varchar | Current stage: `Created` → `PickConfirmed` (source WMS has packed the goods) → `InTransit` (driver en route) → `Received` (destination warehouse confirmed arrival) → `Completed` (stock updated at both locations). |
| `tracking_id` | varchar | The TMS tracking code for the shipment carrying the transferred goods. Set when TMS dispatch is triggered. |
| `created_at` / `updated_at` | timestamptz | Standard audit timestamps. |
| `updated_by` | varchar | Who last updated this record. |

---

### `transfer_order_lines` — individual products in a transfer

One row per product being moved in the transfer.

| Column | Type | Plain meaning |
|---|---|---|
| `to_line_id` | UUID PK | Unique ID for this line. |
| `transfer_order_id` | UUID FK | The transfer order this line belongs to. |
| `sku` | varchar | Product code being transferred. |
| `requested_qty` | int | How many units the destination store requested. |
| `transferred_qty` | int | How many units were actually packed at the source. May be less than `requested_qty` if the source store was short. Updated when WMS confirms pick. |
| `confirmed_at` | timestamptz | When WMS at the source confirmed these items were picked and packed ready to ship. |

---

### `damaged_goods_receipts` — damaged packages returned to the warehouse

When a delivery driver cannot complete a delivery due to a damaged package, they bring it back to the warehouse. This table tracks the check-in and processing of that damaged return.

| Column | Type | Plain meaning |
|---|---|---|
| `damaged_receipt_id` | UUID PK | Unique ID for this damaged goods receipt. |
| `order_id` | UUID FK | The original customer order the damaged package came from. |
| `tracking_id` | varchar | The TMS tracking code of the package being returned. Used to link back to the delivery event. |
| `status` | varchar | Processing stage: `Received` (driver brought it back) → `Inspected` (staff checked each item) → `PutAway` (items shelved, repaired, or disposed) → `Closed` (record complete). |
| `received_at` | timestamptz | When the driver returned the damaged package to the warehouse dock. |
| `put_away_at` | timestamptz | When all items were routed to their final destination (shelf, repair bay, or disposal). |
| `updated_by` | varchar | Who last updated this record. |

---

### `damaged_goods_items` — individual items inside a damaged package

One row per item inspected from the returned damaged package.

| Column | Type | Plain meaning |
|---|---|---|
| `item_id` | UUID PK | Unique ID for this item record. |
| `damaged_receipt_id` | UUID FK | The damaged goods receipt this item belongs to. |
| `sku` | varchar | Product code of the item. |
| `condition` | varchar | Outcome of the inspection: `Resellable` (put back on the shelf — stock is restored), `Repairable` (needs repair before it can be sold), `Dispose` (written off — triggers insurance or cost-of-goods adjustment). |
| `sloc` | varchar | Where the item was sent after inspection: a shelf location code, a repair bay code, or a disposal area code. |
| `confirmed_at` | timestamptz | When warehouse staff confirmed the inspection result and routing for this item. |
