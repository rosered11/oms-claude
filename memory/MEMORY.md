- [Project Architecture](project_architecture.md) — Tech stack, scale, deployment, key architectural constraints and decisions for the OMS

- [Outbox Dispatch Patterns](outbox_dispatch_patterns.md) — Two dispatch paths: `OutboxAdapterService.Dispatch` (full path: HTTP call + OutboxDispatchLog + timeline, required for outbox monitor visibility) vs `ApiResult.BuildOutboxEvents` (timeline annotation only, no HTTP call, no dispatch log); always use the service for any event that must reach an external system
- [Breaking Changes May 2026](breaking_changes_2026_05.md) — State machine simplified (BookingConfirmed/Invoiced/Paid removed); payment_flow changed from bool to string; monetary values changed from satang to baht; GET /branches/nearby removed; ApiResult.DispatchOutbox renamed to BuildOutboxEvents

- [Payment Agent — STS Idempotency](.claude/agents-memory/payment-agent/decision_sts_idempotency.md) — All STS/WMS/TMS webhook idempotency enforced via DB-level UNIQUE constraints (not application pre-checks); covers credit notes, invoices, and order webhook logs
- [Payment Agent — Return Refund Flow](.claude/agents-memory/payment-agent/decision_return_refund_flow.md) — Refund amount = SUM(unit_price × quantity) from return_items; put-away-confirmed is the single atomic trigger for all return/refund writes; no re-authorisation; partial return state machine: Requested → PutAway → Refunded
- [Payment Agent — STS/Gateway Error Mappings](.claude/agents-memory/payment-agent/mapping_sts_gateway_errors.md) — Maps `payment.invoices.status` and `order_payments.status`/`payment_method` to outbox payload fields (`is_success`, `payment_status`, `payment_method`, `tendor`) for Gateway, TMS, and WMS integrations
- [Convention: No Hardcoded Data](convention_no_hardcoded_data.md) — Never embed data in code or JSON files; e2e test data must be created via API calls (POST to endpoints in beforeEach/before), not by seeding JSON or patching in-memory state
